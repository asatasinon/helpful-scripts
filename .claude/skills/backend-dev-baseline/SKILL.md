# Backend Dev Baseline

本 skill 用于在 `baby-growth-record-book` 仓库内执行后端开发时，保证实现和架构文档一致。

先阅读 [references/backend-constraints.md](./references/backend-constraints.md)，再开始编码。

## 适用范围
- 核心 API（auth/family/baby/event/admin）
- 派生能力（summary/analytics/notify/jobs）
- AI 与导出相关后端能力
- 数据库模型、查询与迁移

## 默认工作流

1. 先锁定职责边界。
   确认当前改动属于 `T06`（核心域）还是 `T07`（派生域），避免在一个任务里混改所有模块。

2. 先对契约与模型，再写实现。
   同步核对 `D03 + S01`（接口）与 `D04 + D05 + S02`（实体/DDL），先解决口径冲突再编码。

3. 先写权限和隔离，再写业务细节。
   所有读写链路先检查 `family_id` 归属、成员角色和资源可见性，再进入业务处理。

4. 先保证可追踪性，再做性能优化。
   关键写操作补齐审计与错误码，保证排障可追踪；性能优化放在正确性之后。

5. 变更后做链路自检。
   对照 `Q01` 检查登录、事件写入、编辑删除重算、多家庭隔离、导出重试、AI 兜底。

## 实现准则

## 依赖与工程
- Python 环境、依赖安装和命令执行统一使用 `uv`。
- 不混用 `pip`、`poetry`、`pipenv`。
- API 与 Worker 共享核心领域模型和约束，避免各自维护“平行规则”。

## 接口与错误处理
- 所有 API 按 `/api/v1` 组织，并保持统一响应壳 `{ code, message, data }`。
- API 边界统一使用字符串 ID（`^[0-9]+$`）；通过 Pydantic validator/serializer 实现集中转换，禁止业务层散落转换。
- 错误码只使用约定集合，不新增“临时字符串错误码”。
- 写接口在重试场景支持 `X-Idempotency-Key` 并保证幂等语义。

## 权限、隔离与审计
- 所有业务数据必须绑定 `family_id` 并显式校验归属。
- `viewer` 不可写，角色判断要在服务端落地，不依赖前端。
- 事件删除采用软删除；编辑、删除、角色变更等关键操作写 `operation_logs`。

## 事件与异步派生
- 统一事件写入后，投递 `task_jobs` 触发聚合补算。
- 编辑涉及时间变化时，旧日期和新日期都要重算。
- 趋势优先读取 `metric_snapshots`，缺失时允许回源计算。

## AI 与导出
- AI 只基于提供数据回答，输出必须包含时间窗口与免责声明。
- 模型不可用时返回 `AI_SERVICE_UNAVAILABLE`，不得让主业务链路崩溃。
- 导出任务要有状态机（pending/running/succeeded/failed）和失败原因。

## 数据库与一致性
- 主键统一 `BIGINT`；时间字段统一使用 UTC 毫秒时间戳（`BIGINT`）。
- 保持关键唯一键、外键和 `CHECK` 约束不被代码绕过。
- 查询优先利用既有索引组合，避免在高频路径制造全表扫描。

## 输出格式（执行结果回复）
完成开发后，输出建议包含：
1. 改动模块与文件清单
2. 契约/模型/DDL 对齐说明
3. 权限隔离与审计覆盖说明
4. 自测结果与剩余风险（含跨任务依赖）

## 触发示例

**示例 1**
输入：`补齐事件编辑后的重算任务和失败重试`  
动作：触发本 skill，按事件主写入 + `task_jobs` + 汇总补算链路实现。

**示例 2**
输入：`加家庭成员角色变更接口，并处理只读成员权限`  
动作：触发本 skill，先校验契约与角色模型，再落权限和审计。

**示例 3**
输入：`AI 问答在无数据时不要胡答，并统一错误码`  
动作：触发本 skill，落实 AI 上下文兜底与 `AI_SERVICE_UNAVAILABLE`/数据不足分支。
